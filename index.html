02 Getting Started

Quick Start

Option A: Docker (Recommended - 5 Minutes)

The fastest way to get started with all services pre-configured including PostgreSQL, Redis, Django, Celery, Flower monitoring, and Nginx.

bash ‚Äî Complete Docker setup
# Clone or create project directory
$ mkdir my-django-project
$ cd my-django-project

# Copy all project files (.env.example, Dockerfile, docker-compose.yml, etc.)

# Create environment configuration
$ cp .env.example .env

# Start all services (PostgreSQL, Redis, Django, Celery, Flower, Nginx)
$ docker-compose up -d

# Watch the logs to see migration and setup progress
$ docker-compose logs -f web

# Access services once ready
API: http://localhost:8000
Admin: http://localhost:8000/admin (admin / admin123)
API Docs: http://localhost:8000/api/docs/ (Swagger UI)
ReDoc: http://localhost:8000/api/redoc/
Flower: http://localhost:5555 (Celery monitoring)
Option B: Local Development (10 Minutes)

Setup Python virtualenv with local PostgreSQL and Redis for development. Requires manual service startup but gives more control.

bash ‚Äî Local development setup
# Create and activate virtual environment with Python 3.13
$ python3.13 -m venv venv
$ source venv/bin/activate  # Linux/Mac
# or on Windows:
# venv\Scripts\activate

# Upgrade pip and install dependencies
$ pip install --upgrade pip setuptools wheel
$ pip install -r requirements.txt

# Setup environment file
$ cp .env.example .env
# Edit .env to set DB_HOST=localhost, REDIS_URL=redis://localhost:6379/0

# Run database migrations
$ python manage.py migrate

# Create superuser for admin
$ python manage.py createsuperuser

# Run development server
$ python manage.py runserver

# In separate terminal: Start Celery worker
$ celery -A project worker --loglevel=info

# In another terminal: Start Celery beat scheduler (optional)
$ celery -A project beat --loglevel=info --scheduler django_celery_beat.schedulers:DatabaseScheduler

# In another terminal: Start Flower monitoring (optional)
$ celery -A project flower --port=5555
üí° Tip

Use Docker for fastest setup with all services pre-configured and isolated. Local development works great if you only need Django without Celery/async tasks. Docker Compose is recommended for production-like testing.

PostgreSQL Database Setup

Create a dedicated database and user for your Django project with proper permissions and configuration.

bash ‚Äî Create PostgreSQL database and user
# Access PostgreSQL as superuser
$ sudo -u postgres psql
# or on macOS with Homebrew:
$ psql postgres

# Inside psql shell, run these commands:
CREATE DATABASE project_db;
CREATE USER project_user WITH PASSWORD 'secure_password_here';
ALTER ROLE project_user SET client_encoding TO 'utf8';
ALTER ROLE project_user SET default_transaction_isolation TO 'read committed';
ALTER ROLE project_user SET default_transaction_deferrable TO on;
ALTER ROLE project_user SET timezone TO 'UTC';
GRANT ALL PRIVILEGES ON DATABASE project_db TO project_user;
\q

cp .env.example .env then edit .env with local PostgreSQL and Redis settings

Run Database Migrations

python manage.py migrate

Verify: Should complete without errors

Collect Static Files (Optional)

python manage.py collectstatic --noinput

Only needed for production or testing static file serving

Terminal 1: python manage.py runserver

Terminal 2 (for async tasks): celery -A project worker --loglevel=info --concurrency=4

Terminal 3 (for scheduled tasks): celery -A project beat --loglevel=info

Key Design Principles

Modularity: Settings split by environment (local/production), requirements split by use case (base/dev/prod), apps organized by domain.

Reusability: common/ directory provides base models, serializers, viewsets, permissions for DRY code across all apps.

Scalability: Can add new apps (e.g., api/v2/, analytics/) without affecting existing code. Horizontal scaling with multiple workers.

Testing: tests/ directory with pytest fixtures and factories for easy, comprehensive testing of all components.

Deployment Ready: Docker configuration, environment-based settings, static file handling, logging, and monitoring all built-in.

üí° Tip

Start with the minimal structure and expand as needed. Don't create apps you don't use. Follow Django best practices: one app per feature/domain, reuse code via common/, keep views thin by moving logic to models/serializers.

Modular Settings Structure (NEW in 2025)

Django 6.0 project now uses modular settings for better maintainability, environment-specific configuration, and cleaner code organization.

python ‚Äî settings/__init__.py - Environment loader
"""Settings module with environment-based loading"""
import os

# Determine which settings module to load based on DJANGO_ENVIRONMENT
environment = os.getenv('DJANGO_ENVIRONMENT', 'local')

if environment == 'production':
    from .production import *  # noqa
elif environment == 'development':
    from .local import *  # noqa
else:
    from .local import *  # noqa

Security Hardening

Pre-Deployment Security Checklist

Critical Settings

  • Set DEBUG=False in production settings
  • Generate secure SECRET_KEY (minimum 50 random characters)
  • Configure ALLOWED_HOSTS to your domain(s) only
  • Setup SSL/HTTPS certificates (Let's Encrypt recommended)
  • Enable SECURE_SSL_REDIRECT to redirect HTTP to HTTPS
  • Set SESSION_COOKIE_SECURE=True for HTTPS only
  • Set CSRF_COOKIE_SECURE=True for HTTPS only
  • Enable SECURE_HSTS with 1 year expiration
  • Configure Content Security Policy (CSP) headers
  • Set X-FRAME-OPTIONS to DENY (prevent clickjacking)

Database & Access

  • Use strong database passwords (minimum 16 characters)
  • Restrict database access to application servers only
  • Enable database encryption at rest
  • Setup automated database backups
  • Test database restore procedures
  • Use different users for development/production databases
  • Enable query logging for security audits
  • Setup read-only replicas for backups

Monitoring & Logging

  • Setup error tracking (Sentry recommended)
  • Configure centralized logging (ELK, CloudWatch)
  • Monitor application health and performance
  • Setup alerts for errors and anomalies
  • Log all authentication attempts
  • Monitor resource usage (CPU, memory, disk)
  • Setup log rotation and retention policies
  • Regular security audit logs review

Celery & Async Tasks

What is Celery?

Celery is a distributed task queue library that allows Django to execute long-running operations asynchronously. Instead of blocking a web request while sending emails, processing images, or running reports, Celery queues the task and processes it in the background.

Creating Celery Tasks
python ‚Äî accounts/tasks.py - Celery tasks
from celery import shared_task, current_task
from django.core.mail import send_mail
from django.contrib.auth import get_user_model
import logging

logger = logging.getLogger(__name__)
User = get_user_model()

@shared_task(bind=True, max_retries=3, default_retry_delay=60)
def send_welcome_email(self, user_id: str, email: str):
    """Send welcome email to new user with retry logic"""
    try:
        user = User.objects.get(id=user_id)
        subject = 'Welcome to Project!'
        message = f'Hello {user.first_name}, welcome aboard!'
        
        send_mail(
            subject,
            message,
            'noreply@example.com',
            [email],
            fail_silently=False,
        )
        logger.info(f'Welcome email sent to {email}')
        return {'status': 'success', 'email': email}
    except User.DoesNotExist:
        logger.error(f'User {user_id} not found')
        return {'status': 'error', 'message': 'User not found'}
    except Exception as exc:
        logger.error(f'Error sending email: {str(exc)}')
        # Retry with exponential backoff
        raise self.retry(exc=exc)
Calling Celery Tasks from Views
python ‚Äî accounts/viewsets.py - Call async tasks
from rest_framework import viewsets, status
from rest_framework.response import Response
from .tasks import send_welcome_email
from .serializers import UserSerializer

class UserViewSet(viewsets.ModelViewSet):
    """User API with async welcome email"""
    
    def create(self, request, *args, **kwargs):
        serializer = self.get_serializer(data=request.data)
        serializer.is_valid(raise_exception=True)
        user = serializer.save()
        
        # Queue email sending asynchronously
        send_welcome_email.delay(str(user.id), user.email)
        
        return Response(
            {'message': 'User created. Welcome email queued.'},
            status=status.HTTP_201_CREATED
        )

    # Check cache first

Deployment Strategies

WSGI vs ASGI Servers

Gunicorn (WSGI)

  • Synchronous request handling
  • Perfect for traditional REST APIs
  • Simple, battle-tested, stable
  • Multiple worker processes
  • Best for pure Django + DRF

Use when: Building APIs without WebSockets or real-time features

Uvicorn (ASGI)

  • Asynchronous request handling
  • WebSocket support (Channels)
  • Modern, fast, lightweight
  • Worker process management
  • Better for real-time apps

Use when: Building with Channels or needing async views

Nginx Reverse Proxy

Nginx Role in Production

Nginx serves as a reverse proxy that sits between client requests and your Django application. It handles SSL/HTTPS termination, static file serving, compression, rate limiting, and acts as a load balancer.

nginx ‚Äî Production Nginx configuration
upstream django {
    server web:8000;
}

server {
    listen 80;
    server_name yourdomain.com www.yourdomain.com;
    
    # Redirect HTTP to HTTPS
    return 301 https://$server_name$request_uri;
}

server {
    listen 443 ssl http/2;
    server_name yourdomain.com www.yourdomain.com;
    
    # SSL Certificates (Let's Encrypt)
    ssl_certificate /etc/letsencrypt/live/yourdomain.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/yourdomain.com/privkey.pem;
    
    # Modern TLS configuration
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;
    ssl_prefer_server_ciphers on;
    
    # Security headers
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-Frame-Options "DENY" always;
    add_header X-XSS-Protection "1; mode=block" always;
    
    # Gzip compression
    gzip on;
    gzip_types text/plain text/css application/json application/javascript;
    gzip_vary on;
    gzip_min_length 1000;
    
    # Static files (CSS, JS, images)
    location /static/ {
        alias /app/staticfiles/;
        expires 1y;
        add_header Cache-Control "public, immutable";
    }
    
    # Media files (user uploads)
    location /media/ {
        alias /app/media/;
        expires 7d;
    }
    
    # Proxy requests to Django
    location / {
        proxy_pass http://django;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_redirect off;
        
        # Timeouts
        proxy_connect_timeout 60s;
        proxy_send_timeout 60s;
        proxy_read_timeout 60s;
    }
}

Version Updates & Migration

Django 4.2 ‚Üí 6.0 Migration Path

This guide covers Django 6.0 (latest). If migrating from Django 4.2, review breaking changes and follow the upgrade path through Django 5.0 and 5.2.

Breaking Changes in Django 6.0

  • Python < 3.10 no longer supported
  • PostgreSQL < 13 no longer supported
  • Django admin interface redesigned
  • ORM query optimizations may change query patterns
  • ASGI support improved (better async handling)
  • Signals and middleware processing improved
