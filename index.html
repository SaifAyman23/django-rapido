01 Overview

What's inside

Production‑ready foundation – REST API (DRF 3.16) with JWT, WebSockets via Channels 4.1, background tasks with Celery 5.6, modern admin (Unfold 0.42), and a complete Docker setup.

Modular and scalable – environment‑based settings (local/production), reusable common app, and domain‑driven app structure.

Dev experience – pytest, black, isort, flake8, mypy, pre‑commit, Makefile shortcuts.

REST API

DRF + drf‑spectacular, OpenAPI 3, Swagger UI, ReDoc, JWT auth (SimpleJWT).

Async & Real‑time

Channels 4.1 WebSocket, Redis channel layer, Celery 5.6 worker/beat, Flower monitoring.

Docker Compose

7 services: web, db, redis, celery_worker, celery_beat, flower, nginx – one command to start.

Security hardened

Env‑based secrets, HTTPS redirect, secure cookies, CORS, and check --deploy ready.

03 Quick Start

Docker setup (recommended)

bash – 5 minutes to running 
# Clone (or create your project)
$ git clone <repository-url>
$ cd project

# Copy environment template
$ cp .env.example .env

# Start all services (db, redis, web, celery, flower, nginx)
$ docker-compose up -d

# Access the running app:
#   http://localhost:8000
#   http://localhost:8000/admin   (admin / admin123 after createsuperuser)
#   http://localhost:8000/api/docs/
#   http://localhost:5555          (Flower)
Local setup (without Docker)
bash – virtualenv + local services 
# Create & activate virtual environment
$ python -m venv venv
$ source venv/bin/activate   # or venv\Scripts\activate

# Install dependencies
$ pip install -r requirements.txt

# Configure environment (edit DB_HOST=localhost etc.)
$ cp .env.example .env

# Run migrations
$ python manage.py migrate

# Start development server
$ python manage.py runserver

# In another terminal: start Celery worker
$ celery -A project worker --loglevel=info
💡 Tip Docker setup is recommended – it starts PostgreSQL, Redis, Nginx and all services with one command.

05 Configuration

Environment & settings

Environment files
File Purpose Git
.env.example template with safe defaults ✅ committed
.env base configuration ❌ ignored
.env.local local overrides ❌ ignored
.env.production production secrets ❌ ignored
Settings loading order
  1. DJANGO_ENVIRONMENT determines environment (default: 'local')
  2. .env is loaded (base)
  3. environment‑specific file (.env.local or .env.production) overrides
  4. Python settings (local.py / production.py) are imported

09 Security

Pre‑flight checklist

Verify before production deploy

  • DEBUG=False in production
  • SECRET_KEY long random string (not default)
  • ALLOWED_HOSTS contains your domain(s)
  • HTTPS enabled with valid SSL certificates
  • SECURE_SSL_REDIRECT=True
  • SESSION_COOKIE_SECURE=True
  • CSRF_COOKIE_SECURE=True
  • Database & Redis passwords strong and unique
  • Run python manage.py check --deploy
10 Deploy

Production deployment

Using Docker Compose (on server)
bash – production start 
# On production server
$ cd /app
$ docker-compose -f docker-compose.yml up -d

# Services started: web (gunicorn), nginx (80/443), db, redis, celery_worker, celery_beat, flower
# Make sure .env.production is used (or set DJANGO_ENVIRONMENT=production)
Manual deployment (without Docker)
bash – gunicorn + systemd 
# Collect static & migrate
$ python manage.py collectstatic --noinput
$ python manage.py migrate

# Start gunicorn
$ gunicorn project.wsgi:application \
    --bind 0.0.0.0:8000 \
    --workers 4 \
    --max-requests 1000 \
    --timeout 120
