02 Getting Started

Quick Start

Option A: Docker (Recommended - 5 Minutes)

The fastest way to get started with all services pre-configured including PostgreSQL, Redis, Django, Celery, Flower monitoring, and Nginx.

bash ‚Äî Complete Docker setup
# Clone or create project directory
$ mkdir my-django-project
$ cd my-django-project

# Copy all project files (.env.example, Dockerfile, docker-compose.yml, etc.)

# Create environment configuration
$ cp .env.example .env

# Start all services (PostgreSQL, Redis, Django, Celery, Flower, Nginx)
$ docker-compose up -d

# Watch the logs to see migration and setup progress
$ docker-compose logs -f web

# Access services once ready
API: http://localhost:8000
Admin: http://localhost:8000/admin (admin / admin123)
API Docs: http://localhost:8000/api/docs/ (Swagger UI)
ReDoc: http://localhost:8000/api/redoc/
Flower: http://localhost:5555 (Celery monitoring)
Option B: Local Development (10 Minutes)

Setup Python virtualenv with local PostgreSQL and Redis for development. Requires manual service startup but gives more control.

bash ‚Äî Local development setup
# Create and activate virtual environment with Python 3.13
$ python3.13 -m venv venv
$ source venv/bin/activate  # Linux/Mac
# or on Windows:
# venv\Scripts\activate

# Upgrade pip and install dependencies
$ pip install --upgrade pip setuptools wheel
$ pip install -r requirements.txt

# Setup environment file
$ cp .env.example .env
# Edit .env to set DB_HOST=localhost, REDIS_URL=redis://localhost:6379/0

# Run database migrations
$ python manage.py migrate

# Create superuser for admin
$ python manage.py createsuperuser

# Run development server
$ python manage.py runserver

# In separate terminal: Start Celery worker
$ celery -A project worker --loglevel=info

# In another terminal: Start Celery beat scheduler (optional)
$ celery -A project beat --loglevel=info --scheduler django_celery_beat.schedulers:DatabaseScheduler

# In another terminal: Start Flower monitoring (optional)
$ celery -A project flower --port=5555
üí° Tip

Use Docker for fastest setup with all services pre-configured and isolated. Local development works great if you only need Django without Celery/async tasks. Docker Compose is recommended for production-like testing.

Prerequisites & System Setup

PostgreSQL Database Setup

Create a dedicated database and user for your Django project with proper permissions and configuration.

bash ‚Äî Create PostgreSQL database and user
# Access PostgreSQL as superuser
$ sudo -u postgres psql
# or on macOS with Homebrew:
$ psql postgres

# Inside psql shell, run these commands:
CREATE DATABASE project_db;
CREATE USER project_user WITH PASSWORD 'secure_password_here';
ALTER ROLE project_user SET client_encoding TO 'utf8';
ALTER ROLE project_user SET default_transaction_isolation TO 'read committed';
ALTER ROLE project_user SET default_transaction_deferrable TO on;
ALTER ROLE project_user SET timezone TO 'UTC';
GRANT ALL PRIVILEGES ON DATABASE project_db TO project_user;
\q
Redis Configuration

Redis setup for caching, sessions, and Celery message brokering. Optional password protection for production.

bash ‚Äî Test Redis connection

# In another terminal, test connection
$ redis-cli ping
# Should return: PONG

# Test more commands

Environment Configuration

Create .env file with all necessary environment variables for local development or production.

bash ‚Äî Create .env file

$ nano .env  # Edit with your values

# Key variables for local development:
DJANGO_ENVIRONMENT=local
DEBUG=True

DB_ENGINE=django.db.backends.postgresql

Never commit .env files to version control. Use .env.example as a template. Generate unique SECRET_KEY for production using: python -c "from django.core.management.utils import get_random_secret_key; print(get_random_secret_key())"

python manage.py collectstatic --noinput

Terminal 1: python manage.py runserver

Terminal 2 (for async tasks): celery -A project worker --loglevel=info --concurrency=4

Key Design Principles

Modularity: Settings split by environment (local/production), requirements split by use case (base/dev/prod), apps organized by domain.

Reusability: common/ directory provides base models, serializers, viewsets, permissions for DRY code across all apps.

Scalability: Can add new apps (e.g., api/v2/, analytics/) without affecting existing code. Horizontal scaling with multiple workers.

Testing: tests/ directory with pytest fixtures and factories for easy, comprehensive testing of all components.

Deployment Ready: Docker configuration, environment-based settings, static file handling, logging, and monitoring all built-in.

üí° Tip

Start with the minimal structure and expand as needed. Don't create apps you don't use. Follow Django best practices: one app per feature/domain, reuse code via common/, keep views thin by moving logic to models/serializers.

Settings Configuration

Modular Settings Structure (NEW in 2025)

Django 6.0 project now uses modular settings for better maintainability, environment-specific configuration, and cleaner code organization.

python ‚Äî settings/__init__.py - Environment loader
"""Settings module with environment-based loading"""
import os

# Determine which settings module to load based on DJANGO_ENVIRONMENT
environment = os.getenv('DJANGO_ENVIRONMENT', 'local')

if environment == 'production':
    from .production import *  # noqa
elif environment == 'development':
    from .local import *  # noqa
else:
    from .local import *  # noqa

  • Development environment overrides

  • Console email backend
