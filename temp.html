Careerly Flows Documentation

Complete system design covering seven core flows: authentication, home page, job discovery, notifications, user profiles, subscriptions, and monitoring.

Overview

This document specifies all Careerly flows in comprehensive detail. Each flow covers business logic, data models, API endpoints, technical architecture, and detailed developer notes for backend, frontend, mobile, and AI engineers. All diagrams use Mermaid and are fully interactive.

The seven flows are designed to work together as an interconnected system where each flow depends on data and state from others. The documentation is structured as: analysis (what happens), diagrams (how it flows), technical specs (how to build it), and implementation notes (how to actually code it).

CAREERLY-001: Authentication Flow

Flow ID: CAREERLY-001
Status: Core flow
Related: CAREERLY-005 (Profile), CAREERLY-006 (Subscription)

Overview

This flow covers all three authentication routes for Careerly: registration, login, and password reset. It exists to securely onboard new job seekers and return authenticated users to the platform. The business value is establishing a verified user identity tied to a resume and preferences, which every other feature in the platform depends on.

Routes

Route 1: Register

  • Job Seeker navigates to Register screen and enters email, password, confirm password
  • System validates all fields (format, strength, match)
  • System checks if email is already registered
  • System creates user record with status = UNVERIFIED
  • System generates 6-digit OTP, stores it with 10-minute expiry
  • Job Seeker enters OTP and system verifies it
  • System marks user status = VERIFIED, generates auth token
  • Job Seeker selects job preferences (job titles, locations, job types)
  • Job Seeker uploads resume (PDF, required)
  • System validates and saves resume, dispatches Celery Task to parse it
  • Job Seeker is redirected to Home page with JWT tokens issued

Route 2: Login

  • Job Seeker enters email and password
  • System looks up email, verifies password hash
  • System checks user status (must be ACTIVE or SETUP_INCOMPLETE)
  • System issues JWT access token + refresh token
  • Job Seeker is redirected to Home page

Route 3: Password Reset

  • Job Seeker enters email address
  • System generates 6-digit OTP and sends via email
  • Job Seeker enters OTP and system verifies it
  • System issues short-lived password reset token (15 minutes, single use)
  • Job Seeker enters new password twice, system validates and saves
  • System invalidates all existing sessions (blacklists JWT tokens)
  • Job Seeker is redirected to Profile page (logged in automatically)

Developer Notes

Backend Developer (Django)

Use AbstractBaseUser + BaseUserManager for custom User model. Use djangorestframework-simplejwt for JWT with ACCESS_TOKEN_LIFETIME = 15 minutes and REFRESH_TOKEN_LIFETIME = 30 days. Implement token blacklist on logout and password reset. Use secrets.randbelow(1000000) for OTP generation, hash with SHA-256 before storing. Dispatch Django Tasks via transaction.on_commit() for OTP email sending. Implement rate limiting at both model level (attempts counter) and API level (django-ratelimit). Use MultiPartParser for setup endpoint handling. Resume parsing: use django-storages with S3 or local FileField. Post-save signal on UserPreferences: check if user also has resume, then set user.status = ACTIVE.

Frontend Developer (React)

Register screen: email, password, confirm password fields with strength indicator. OTP screen: 6 individual digit input boxes with auto-advance. Preferences + Resume screen: multi-select for job titles (searchable dropdown), checkbox group for job types, drag-and-drop file upload zone. Login screen: email + password with "Forgot password?" link. Password reset — New Password screen: 2 fields with strength indicator. Store JWT access token in memory, refresh token in HttpOnly cookie or secure storage. Redirect based on user status: if SETUP_INCOMPLETE, redirect to setup screen.

Mobile Developer (Flutter)

OTP input: Row of 6 TextField widgets, max_length=1, keyboardType=number. Resume upload: use file_picker package, filter to PDF only. Token storage: use flutter_secure_storage for both tokens. Implement HTTP interceptor for 401 handling and silent refresh. Register deep link: careerly://auth/verify-otp to direct users from OTP emails. Use pushReplacement after login/setup so users can't navigate back to auth screens.

AI Engineer

On successful setup (resume uploaded + preferences saved), dispatch Celery task to parse resume. Input: resume file (PDF), user ID. Output: extracted fields — job titles, skills, years of experience, education level as structured JSON on Resume model. This parsed data feeds job recommendations. Latency: async — show "We're setting up your recommendations" on first home page visit if parsing still in progress. Fallback: if parsing fails, flag resume as parse_status=FAILED and retry later. Recommendations fall back to preferences-only matching until re-parsed.

CAREERLY-002: Home Page Flow

Flow ID: CAREERLY-002
Status: Core flow
Related: CAREERLY-001 (Auth), CAREERLY-003 (Job Detail)

Overview

The home page is the core discovery surface of Careerly. It serves two sections: a personalized recommendations section (driven by the user's resume and preferences) and a general job listing section (all scraped jobs, paginated). Jobs are scraped every 15 minutes via a Celery Beat task that pulls from Indeed, LinkedIn, Glassdoor, and Naukri. Each job in the feed is tagged as "new" if it was added in the last 2 hours, or "viewed" if the user has previously opened that job's detail page.

Sub-flows

Sub-flow A: Scraper Pipeline (background, runs every 15 minutes)

  • Celery Beat triggers scrape_all_platforms task every 15 minutes
  • Celery Task runs scrapers for Indeed, LinkedIn, Glassdoor, Naukri in parallel (Celery group)
  • Each scraper fetches latest job listings, normalizes data
  • System checks if jobs already exist by external_id + platform
  • New jobs: inserts with status=ACTIVE, scraped_at=now
  • Duplicates: skips insertion, updates last_seen_at only
  • System notifies users whose preferences/resume keywords match newly added jobs
  • System logs scrape cycle results per platform

Sub-flow B: Home Page Load (user-triggered)

  • Job Seeker lands on home page (authenticated)
  • System checks if resume parsing is still in progress — shows loading state if so
  • System checks Redis cache for this user's recommendations
  • Cache hit: returns cached recommendations immediately
  • Cache miss: triggers recommendation generation asynchronously, returns loading state
  • System returns first page of general job listing (paginated, 20 per page, sorted by scraped_at DESC)
  • System checks user's JobView history and applies tags: "viewed" if viewed before, "new" if added in last 2 hours AND not viewed
  • Job Seeker sees recommendations section and job listing with tags

Sub-flow C: Recommendation Generation (async)

  • System fetches user's parsed resume data and preferences
  • System queries jobs DB for jobs matching user's profile
  • AI/ML Service scores and ranks matched jobs by suitability (resume embedding vs job embedding)
  • Takes top N recommendations (default 10)
  • Stores in Redis cache with TTL 30 minutes
  • Returns recommendations to home page response

Developer Notes

Backend Developer (Django)

Celery Beat: schedule scrape_all_platforms every 15 minutes via crontab(minute='*/15'). Use celery.group to run all 4 platform scrapers in parallel. Company upsert first, then Job upsert. For each scraped job: (1) get_or_create Company by name__iexact, (2) update_or_create Job by (external_id, platform), only update last_seen_at on re-scrape. Never overwrite existing non-null Job fields. All nullable fields store null, never empty string. Use select_related('company') and only() for job listing query. Compute job tags from pre-fetched viewed_ids set, not per-job DB queries. Daily Celery Beat task: mark jobs EXPIRED if last_seen_at < now-48h, ARCHIVED if last_seen_at < now-7d. Cache recommendation results in Redis with key recommendations:user:{id}, TTL 30 min. Invalidate when new matching jobs are scraped.

Frontend Developer (React)

Home page: two sections — Recommendations (horizontal scroll or grid) and Job Listing (vertical infinite scroll). Each job card: title, company name/logo, location, platform badge, job type, remote badge, "new"/"viewed" tag, date_posted. Recommendations section: skeleton loader. If resume parsing in progress, show banner. Job listing: infinite scroll with 80% trigger for next page load. Read job.tag from API response — "new" green badge, "viewed" grey badge, null nothing. Show platform logo. Salary: show only if min/max available, format as "$X – $Y /year" using interval and currency. Cache home page data with 5-minute stale time (React Query/SWR).

Mobile Developer (Flutter)

Recommendations section: ListView with scrollDirection: Axis.horizontal. Job listing: ListView.builder with lazy loading next page when user reaches last 5 items. Job cards: Card widget with platform icon, title, company, location, tag badge. Tag badges: Chip widget — read job.tag from API. Pull-to-refresh: RefreshIndicator wrapper calling GET /api/v1/home/. If recommendations loading: show Shimmer effect. Store first page in local cache (hive/isar) for offline. Platform icons: bundle as local assets.

AI Engineer

Recommendation generation: given user's resume data (skills list, job titles, experience level), query ACTIVE jobs and score by relevance using embedding similarity. Pre-compute job embeddings using pgvector on PostgreSQL or separate vector store. On recommendation request, compute user's profile embedding and run nearest-neighbor search. Fallback to PostgreSQL full-text search if vector DB unavailable. Cache with 30-minute TTL. Invalidate when new matching jobs arrive.

CAREERLY-003: Jobs Flow

Flow ID: CAREERLY-003
Status: Core flow
Related: CAREERLY-002 (Home Page), CAREERLY-004 (Notifications)

Overview

This flow covers everything that happens after a user selects a job. The job detail page presents scraped and organized data about the job, and offers four actions: save for later, apply now (external redirect), AI Match (resume vs job suitability scoring), and Check Your Resume (general ATS scoring). Both AI features are analysis sessions — they persist to the user's history and contribute to profile stats. The AI Match session is scoped to a specific job; the Check Resume session is general. Both support uploading additional resumes for comparison within the same session, and both render AI responses in markdown.

Routes

Route 1: View Job Detail

  • Job Seeker clicks on a job card
  • System records a JobView for this user + job (upsert)
  • System fetches job details from DB
  • System shows "This job is no longer available" if job doesn't exist or is archived
  • Job Seeker sees job detail page with all available scraped fields
  • System shows notice for fields not available from source platform

Route 2: Save Job

  • Job Seeker clicks the Save button on detail page
  • System checks if job is already saved
  • If already saved: toggles to unsave, removes from saved jobs
  • If not saved: saves the job, button state changes to "Saved"
  • Job Seeker stays on detail page

Route 3: Apply Now

  • Job Seeker clicks "Apply Now" button
  • System retrieves external_url for this job
  • System opens external job page in new tab/in-app browser

Route 4: AI Match

  • Job Seeker clicks "AI Match" button on job detail page
  • System checks if user has resume on file — prompts upload if not
  • System creates new AISession record with type=JOB_MATCH, status=PENDING
  • Job Seeker navigated to AI Match screen
  • System dispatches Celery Task to analyze resume against this job
  • AI/ML Service returns markdown report with: suitability score (0-100), verdict tier, strengths/gaps, top 3 recommended job titles
  • System resolves top 3 recommended titles against jobs DB
  • System updates AISession with score, verdict, report_markdown, top_jobs, status=COMPLETE
  • Job Seeker sees rendered markdown report, score, verdict badge, 3 recommended jobs
  • Job Seeker can upload second resume to compare

Route 5: Check Your Resume (ATS Analysis)

  • Job Seeker clicks "Check Your Resume" button
  • System checks if user has resume on file
  • System creates new AISession record with type=ATS_CHECK, status=PENDING
  • Job Seeker navigated to Check Resume screen
  • System dispatches Celery Task for ATS analysis
  • AI/ML Service returns markdown report with: ATS score (0-100), verdict tier, keyword coverage, formatting recommendations, skills gap suggestions
  • System updates AISession with score, verdict, report_markdown, status=COMPLETE
  • Job Seeker sees rendered markdown report, ATS score, verdict badge
  • Job Seeker can upload second resume to compare ATS scores

Developer Notes

Backend Developer (Django)

GET /api/v1/jobs/{id}/ should upsert JobView inside the view using update_or_create. POST /api/v1/jobs/{id}/save/ should toggle save using get_or_create then delete. AI sessions return 202 Accepted immediately — async via Celery. Celery task run_ai_match: fetch session, fetch resume file, send to AI, parse response, compute verdict (85-100=EXCELLENT, 70-84=GOOD, 50-69=FAIR, 0-49=POOR), resolve top 3 jobs, update session. Set CELERY_TASK_SOFT_TIME_LIMIT=280 and CELERY_TASK_TIME_LIMIT=300 (5 min). Wrap in try/except — on exception, set status=FAILED. Store markdown exactly as returned by AI — do not sanitize. Top 3 job resolution: DB query inside Celery task. Average score computation: use Django Avg aggregation, cache in Redis 10 min. Resume is_default uniqueness: use UniqueConstraint(fields=['user'], condition=Q(is_default=True)). Post-delete signal on Resume: if deleted was default, set latest remaining as default.

Job detail: header (title, company, location, platform badge, save button, apply button), body (description with null notice, skills chips, job type, salary), AI actions (AI Match, Check Resume buttons). Mobile Developer (Flutter)

AI Engineer

CAREERLY-004: Notifications Flow

Flow ID: CAREERLY-004
Status: Core flow
Related: All flows trigger notifications

Overview

Careerly has two categories of notifications: system notifications (triggered by specific user-related events like AI analysis completion or account changes) and new jobs notifications (triggered at the end of each Celery scrape cycle when relevant new jobs are found for a user). System notifications are dispatched as Django Tasks — lightweight, event-driven, tied directly to user actions. New jobs notifications are dispatched from within the Celery scrape pipeline. All notifications are stored in the database for in-app display and marked as read/unread. Push notifications to mobile are delivered via Firebase Cloud Messaging.

CAREERLY-005: Profile Flow

Flow ID: CAREERLY-005
Status: Core flow
Related: CAREERLY-001 (Auth), CAREERLY-003 (AI sessions)

Overview

The profile flow covers both viewing and editing a job seeker's profile. The profile view is a summary dashboard showing the user's personal info, computed stats (average ATS score, average suitability score, total jobs viewed), their uploaded resumes, and a list of jobs they have viewed. The edit profile page allows the user to update their personal information and manage multiple resume files. Biography field supports markdown. Stats are aggregated from AISession and JobView records.

Routes

Route 1: View Profile

  • Job Seeker navigates to profile screen
  • System fetches user data, computed stats, resumes, viewed jobs list
  • System computes stats:
    • Average ATS score: average of all completed ATS_CHECK sessions
    • Average suitability score: average of all completed JOB_MATCH sessions
    • Total jobs viewed: count of JobView records
  • Job Seeker sees profile image, name, position, scores, biography (markdown), resumes list, viewed jobs list

Route 2: Edit Profile

  • Job Seeker taps "Edit Profile"
  • System loads current profile data into form
  • Job Seeker updates: profile image, name, email, phone, position, country, biography
  • System validates all updated fields
  • If email changed: updates email, dispatches Django Task for notification, no re-verification needed
  • System saves updated profile

Route 3: Manage Resumes

  • Job Seeker navigates to resumes section
  • System lists all uploaded resumes (filename, size, upload date, parse status)
  • Job Seeker uploads new resume (PDF or DOCX, max 5MB)
  • System validates file
  • System saves resume, dispatches Celery Task to parse it
  • If first resume, sets as default automatically
  • Job Seeker can set any resume as default
  • Job Seeker can delete resume — if deleting default and others exist, system auto-sets most recent as new default

CAREERLY-006: Subscription Flow

Flow ID: CAREERLY-006
Status: Core flow
Related: CAREERLY-003 (AI features gated)

Overview

Careerly uses a subscription model to govern access to premium features. Every user starts on the FREE plan. Plans define feature flags and usage limits (e.g. number of AI Match sessions per month). The feature configuration is stored in the database and can be changed by admin without code deployment. This flow covers plan assignment, upgrade, and feature gate enforcement. Payment processing is out of scope for this version.

Routes

Route 1: Plan Assignment at Registration

  • System — when new user completes registration, automatically assigns FREE plan
  • System — creates UserSubscription record with plan=FREE, status=ACTIVE, start_date=today

Route 2: View Plans & Upgrade

  • Job Seeker navigates to Subscription screen
  • System fetches all available plans with features and limits
  • Job Seeker sees current plan (highlighted) and upgrade options
  • Job Seeker selects a plan to upgrade to
  • System updates UserSubscription with new plan and start_date=today
  • Job Seeker immediately has access to new plan's features

Route 3: Feature Gate Enforcement

  • Job Seeker attempts to use a feature (e.g. AI Match)
  • System checks user's active plan
  • System checks if feature is enabled for this plan
  • System checks if user has remaining usage for this feature
  • If usage limit reached: blocks action, shows upgrade prompt
  • If allowed: allows action, increments usage counter

Overview

The monitoring flow provides admins with a real-time and historical view of Careerly's health and usage.     • Stat cards: total AI sessions, avg ATS score, avg match score
    • Subscription conversion: free vs paid user ratio
    • Active users per day (7d), scraper success rate per platform
